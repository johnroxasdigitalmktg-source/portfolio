Site Teardown — QADG Motfware (initial)

Scope: public-facing pages reachable without authentication. This is an initial inspection focused on injection risks (XSS/SQLi/command injection) and recommended protections.

What I inspected

  • Homepage: https://qadg.motfware.com/ (content, links, visible controls)
  • `robots.txt` (allowed crawling)
  • OAuth entry point: `/auth/google` (redirect to Google)

Initial findings (evidence)

  • Homepage contains login/register prompts and language selector — obvious input touchpoints (login form, create account flow).
  • OAuth via Google visible; OAuth endpoints may still accept redirect_uri and state parameters — validate these.
  • No public sitemap discovered via `/sitemap.xml` (or fetch failed) — ability to discover all endpoints may be limited.

Likely injection vectors

  • Reflected XSS via query parameters used in UI (language selector, return URLs, error messages).
  • Stored XSS in user-editable fields (profile, referral comments, ambassador program descriptions).
  • SQL injection where backend frameworks or raw queries accept user input without parameterization (registration/login flows).
  • OAuth open redirect if `redirect_uri` is not strictly validated.

Immediate protections to implement (high priority)

  • Server-side input validation: validate type, length, allowed character sets for every parameter.
  • Use parameterized queries / prepared statements for all database access. Replace string interpolation in SQL with bound parameters.
  • Context-aware output encoding: HTML-encode data written into HTML, attribute-encode values in attributes, JS-encode values inserted into scripts, URL-encode in URLs.
  • Strong Content-Security-Policy (CSP): block inline scripts/styles by default, use hashes or nonces for allowed inline code. Example header: 
    Content-Security-Policy: default-src 'self'; script-src 'self' 'nonce-...'; object-src 'none'; base-uri 'self';
  • HTTP-only, Secure, SameSite=strict cookies for session cookies; set short lifetimes for sensitive tokens and rotate tokens on privilege changes.
  • Validate OAuth redirect URIs against an allow-list; reject open redirects and unregistered redirect_uris.
  • Sanitize rich text / HTML input with a whitelist sanitizer (DOMPurify, Bleach, etc.) when storing HTML from users.

Secondary protections (medium priority)

  • Use an ORM or query builder to avoid hand-rolled SQL where possible.
  • Apply output encoding libraries provided by your web framework / template engine rather than manual string concatenation.
  • Rate-limit authentication endpoints and add account lockout thresholds to slow brute-force and automated injection attempts.
  • Enable CSP reporting endpoint (report-uri/report-to) during rollout to learn about blocked attempts.
  • Implement WAF rules for common SQLi/XSS patterns (as a short-term mitigation).

Testing & verification checklist

  • Run automated scanners: OWASP ZAP, Burp Suite (active scan) focused on XSS/SQLi.
  • Fuzz form inputs and parameters (return_url, lang, error) to detect reflections without encoding.
  • Verify prepared statements by auditing code paths for string concatenation into SQL.
  • Use browser devtools to check CSP and cookie flags; verify no inline scripts execute without nonce/hash.
